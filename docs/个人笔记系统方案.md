## 1. 用户背景与痛点

### 需求

想要一个自己的 notebook，方便**各端同步、查看、编辑**。

### 试用过的产品及不满

| 产品 | 评价 |
|---|---|
| 印象笔记 | 太重、收费、设备同步数量有限制、笔记大小有限制 |
| 苹果备忘录 | 常用，但产品形态是"备忘录"不是"笔记本"，太简单 |
| Obsidian | **很喜欢**，尤其是 Claude 插件生态；但 iCloud 同步不方便，手机端打开慢、经常"重新索引"，手机上查看和搜索体验不如印象笔记 |

### 核心诉求（按优先级）

1. **手机上随时查看、搜索笔记**，体验要流畅（当前最大痛点）
2. **移动端也能用 AI**：针对一篇文章深入追问、让 AI 整理整个目录（用户在桌面端通过 Claudian 插件获得此能力，认为是"太强大"的功能，手机上没有很遗憾）
3. 同步要省心，不要 iCloud 那种黑盒
4. 数据自己掌控（本地 Markdown，不被厂商锁定）

### 用户环境

- **Mac**（主力工作机，Obsidian + Claudian 工作流已在使用）
- **iPhone**（主力手机）+ 一台 Android（次要）
- 身在**国内**，访问境外大模型通过**自建中转站**：一台国外低配 VPS，专机专用做 API 中转
- 本机有 **ClashX 代理**：`127.0.0.1:7890`（http/https/socks5 通用）。Claude 执行网络请求失败时应直接使用此代理重试，不必询问用户
- 用户对 Claude Code 工作流熟悉，正在用 Happy（Claude Code 的移动/远程客户端）

## 2. 已拍板的方案：总体架构

**核心认知**：用户真正离不开的不是 Obsidian，而是 **Claude Code 本身**。Obsidian 的价值在于把笔记存为纯 Markdown 文件，让 Claude Code 能直接读写。因此方案围绕"笔记 = 一个 Git 仓库里的 Markdown 文件"展开，各端工具围绕这个真相源分工。

```
┌─────────────┐   Git    ┌──────────────┐
│  Mac        │◄────────►│ GitHub 私有库  │◄────────┐
│  Obsidian   │          │ （笔记仓库 =   │         │
│  + Claudian │          │   唯一真相源） │         │ Git
│  + Obsidian │          └──────────────┘         │
│    Git 插件  │                                   │
└─────────────┘                              ┌────▼─────┐
                                             │ 国外 VPS  │
┌─────────────┐    Happy App                 │ Claude   │──► Anthropic API
│  iPhone     │◄────────────────────────────►│ Code     │    （VPS 在境外，
│  Obsidian   │    （手机干 AI 重活的入口）     │ + 笔记库  │     可直连，无需
│  + Copilot  │                              └──────────┘     经过中转站）
└─────────────┘
```

### 各端分工

| 端 | 工具 | 职责 |
|---|---|---|
| Mac | Obsidian + Claudian（现状不变） | 主力编辑；AI 重活（整理目录、批量加工、抓网页存档） |
| Mac | Obsidian Git 插件 | 自动 commit/push（如每 10 分钟），同步层 |
| GitHub 私有库 | — | 同步 + 版本历史 + 备份，三合一。**若国内推送不稳，挂代理或改用 Gitee/自建 Gitea** |
| 国外 VPS | Claude Code + 笔记仓库 clone | 24/7 在线的 "AI 大脑"，**直连 Anthropic API**（VPS 在境外，无需中转站），低配机器足够（Claude Code 只是 Node CLI，推理在远端） |
| iPhone | Obsidian Mobile + Copilot 插件 | 日常查看、搜索、轻问答（针对单篇追问、全库 Vault QA） |
| iPhone | Happy App | 远程连 VPS 上的 Claude Code，干 AI 重活（不依赖 Mac 开机） |

### 工具选型说明（为什么不选别的）

- **Copilot 插件**（logancyang/obsidian-copilot，v3.3.0+）：官方支持移动端（manifest `isDesktopOnly: false`），支持自定义第三方 API（Provider 选 `3rd party (openai-format)`，填中转站 Base URL + token）。定位是 **RAG 问答 + 写作助手**，不能操作文件。
- **Claudian**（YishenTu/claudian）：把 Claude Code CLI 嵌进 Obsidian，是**能读写移动文件的 agent**，但仅桌面端。比喻：Copilot 是顾问（只动嘴），Claudian 是管家（能动手）。
- **不选思源笔记**：移动端体验虽好，但内置 AI 是问答助手级别，给不了 Claude Code 的 agentic 体验，换了等于降级。
- **不选 Notion/语雀/飞书**：在线文档形态，数据在厂商手里，无本地 AI 插件玩法，且 Notion 国内网络不稳。
- **不用 Obsidian 官方 Sync/iCloud**：Git 对纯 Markdown 是更好的同步层——增量、有版本历史、冲突可见、不黑盒，还顺带解决了备份。

### Copilot 手机端配置要点

路径：`Settings → Copilot → Model → Add Custom Model`

| 配置项 | 填什么 |
|---|---|
| Model Name | 中转站上的模型 ID（需与中转站支持的名称完全一致） |
| Provider | `3rd party (openai-format)` |
| Base URL | 中转站地址，**通常要带 `/v1`** |
| API Key | 中转站 token |

注意事项：
- 填完先点 **Verify** 测通再保存
- 若 Verify 能过但聊天报错，开启设置里的 **Enable CORS**（手机端 WKWebView 常见问题）
- API key 存在系统 Keychain，**桌面和手机需分别填写**
- 中转站若是 new-api / one-api 系，默认就是 OpenAI 兼容接口，可直接用

## 3. 微信文章收藏方案（已确定方向，未实施）

**问题**：微信封闭，文章不能直接分享到外部 App。所有方案第一步统一为：文章页"…"→ **复制链接**。

### 终态方案（路线 A，配合 VPS）

```
微信复制链接 → iOS 快捷指令（一键）→ POST 到 VPS 收件接口
→ 抓正文转 Markdown → git commit 进笔记库 inbox/
→ （可选）Claude Code 顺手做摘要、打标签、建议归档目录
→ 三端同步
```

**关键坑**：微信公众号对海外 IP 有风控，VPS 直接抓 mp.weixin.qq.com 可能返回"环境异常"。**稳妥做法：快捷指令在手机端先抓下网页内容（手机在国内零风控），把 HTML 一起 POST 给 VPS**，VPS 只负责转换和归档。抓取失败时降级为只存标题+链接，提醒用户在 Mac 上补抓。

### 过渡方案（路线 B，零开发先用起来）

iOS 快捷指令：读剪贴板 → 获取 URL 内容 → 提取正文转 Markdown（公众号正文在 `#js_content`，结构规整）→ 写入 Obsidian 库 `inbox/`（Git 迁移后可通过 Working Copy 的 x-callback URL 顺手 commit）。

### 不采用的方案

Cubox/收趣等第三方剪藏 App：多一个数据 silo，与用户离开印象笔记的理由相同，不推荐作为主流程。
